# DeltaCoder v1.2 — Redo Plan (REVISED — Post Dry-Run)

**Goal:** Redo v1.2 SFT + DPO at **32K context** with proper tool-calling data, targeting the BEST 9B for (1) Coding, (2) Tool Calling, (3) Agentic Workflows. Must preserve vision capabilities.

**Architecture:** Unsloth FastVisionModel (text-only LoRA with packing) -> SFT at 32K on H100 -> validate -> DPO -> merge -> GGUF -> test on Kronk -> upload to HuggingFace.

**Tech Stack:** Unsloth (SFT), vLLM nightly (DPO pair gen), TRL (DPO training), ik_llama.cpp (GGUF export)

**Status:** Bootstrap and dry run validated. Full training blocked on cost/duration decision.

---

## What went wrong in the first attempt

1. `normalize_messages()` in `pretokenize.py` corrupted ~65K+ tool-calling examples (converted structured `tool_calls` into raw text)
2. Trained at 8K context (truncated reasoning traces)
3. No validation gate before DPO — wasted money on degraded DPO pairs

## What changed in the redo

1. **32K context** instead of 8K — full reasoning traces preserved
2. **262K training rows** (rebuilt mix: 30.5% nemotron_tool_calling, 19.1% opencoder_reasoning, 15.3% code_feedback, 11.5% swesmith, 11.5% magicoder, 6.5% nemotron_agentic, 5.7% xlam)
3. **Unsloth instead of Axolotl** — Axolotl OOMs at 32K (Liger fused CE doesn't patch VL model), Unsloth handles it internally
4. **FastVisionModel + finetune_vision_layers=False** — preserves vision weights while only training language layers
5. **VLM packing unblock** — Unsloth blocks packing for VLMs; we patch trainer.py to remove the `is_vlm` check (validated in unslothai/unsloth#4160)
6. **Pre-tokenized parquet dataset** — tokenized offline on Romulus (33 min), avoids 4.6hr on-GPU tokenization
7. **Validation gates** — test SFT model before DPO, test on Kronk before uploading

## The Cost Problem

176 hours on H100 at $2.15/hr = **$378 for SFT alone.** Add DPO and it's $400+. This is too expensive for a training run that might need iteration.

### Why it's slow

- **42,976 packed sequences** — packing reduces 262K variable-length conversations into 42,976 fixed 32K-token sequences
- **~15s per forward/backward** — 9.6B param model at 32K context, even with bf16 LoRA
- **batch_size locked at 1** — Qwen3.5 GDN architecture doesn't support batch_size > 1 with sample packing
- **Single GPU** — Unsloth with VLM packing patch is untested on multi-GPU

### Options to Reduce Cost

#### Option 1: Reduce Dataset (RECOMMENDED)
Cut from 262K to ~100-130K most impactful rows.

- **Pros:** ~50% cost reduction (~$180-190), often better quality (less redundancy)
- **Cons:** Need to decide what to cut, risk losing coverage
- **How:** Remove duplicate/low-quality rows, keep highest-quality coding + tool-calling + agentic samples
- **Estimate:** ~100K rows -> ~16K packed examples -> ~67 hours -> ~$144

#### Option 2: Multi-GPU (2-4x H100)
Data parallel across multiple GPUs for linear speedup.

- **Pros:** 2x H100 = ~88 hours, 4x = ~44 hours
- **Cons:** Unsloth + VLM packing patch is untested on multi-GPU. May need to switch to a different training framework (Axolotl FSDP2 — but that OOMs at 32K). Cost per hour is higher (2x H100 ~$4.30/hr, total ~$378 still).
- **Risk:** High — uncharted territory

#### Option 3: Shorter Context (16K)
Reduce max_seq_length from 32K to 16K.

- **Pros:** Roughly 2x speedup (~88 hours, ~$189). Less VRAM.
- **Cons:** Truncates long reasoning traces. User explicitly rejected 8K; 16K may still cut off complex chains.
- **Risk:** Quality degradation on long-form tasks

#### Option 4: QLoRA (4-bit base)
Load base model in 4-bit, train LoRA on top.

- **Pros:** ~2x faster forward passes, much less VRAM (~20GB)
- **Cons:** Known quality degradation vs bf16 LoRA on smaller models. Unsloth supports it but Qwen3.5 GDN layers may have quantization issues.
- **Risk:** Medium — quality tradeoff

#### Option 5: Different Accelerator
Use a different GPU setup that may be cheaper.

- **8x A100 cluster**: Lower per-hour cost, but need framework support for multi-GPU
- **H200**: Faster HBM, slightly faster per-step. But availability and pricing vary.
- **AMD MI300X**: Cheapest $/TFLOP on Vast.ai, but Unsloth doesn't support AMD

#### Option 6: Reduce LoRA Rank
Lower r from 64 to 32 or 16. Fewer trainable params = faster backward pass.

- **Pros:** Maybe 10-20% speedup
- **Cons:** Less capacity, may underfit
- **Risk:** Low-medium

## Infrastructure

### Bootstrap (Validated)
The provisioning script (`v1.2/scripts/provision.sh`) sets up a fresh instance in ~4 minutes:
1. Creates Python 3.12 venv at `/workspace/venv/`
2. Installs Unsloth 2026.3.18 + all dependencies
3. Clones and patches causal-conv1d for SM 9.0 only (H100) — 3 min compile
4. Installs flash-linear-attention
5. Downloads scripts from GitHub (train_unsloth.py, patch_vlm_packing.py)
6. Applies VLM packing unblock patch
7. Pre-downloads Qwen3.5-9B tokenizer/config

**To create a fresh instance:**
```bash
vastai search offers 'gpu_name=H100_SXM num_gpus=1 dph<2.5 reliability>0.95 geolocation in [US,DE,NL,GB,CZ]' --order 'dph' --raw

vastai create instance <OFFER_ID> \
  --image vastai/pytorch:2.10.0-cu128-cuda-12.9-mini-py312-2026-03-26 \
  --env '-e DATA_DIRECTORY="/workspace/"' \
  --disk 80 \
  --ssh --direct

# Then SSH in and run:
curl -fsSL -o /workspace/provision.sh https://raw.githubusercontent.com/danielcherubini/DeltaCoder/main/v1.2/scripts/provision.sh
bash /workspace/provision.sh
```

Note: The `PROVISIONING_SCRIPT` env var does NOT auto-run on the `vastai/pytorch` image. Must run manually.

### Why Unsloth (not Axolotl)
| Aspect | Axolotl | Unsloth |
|--------|---------|---------|
| 32K OOM | YES — Liger fused CE only patches ForCausalLM, not ForConditionalGeneration | NO — handles fused CE internally |
| VLM support | Loads as ForConditionalGeneration, but loss function OOMs | FastVisionModel preserves vision + fused CE works |
| Packing | Works out of the box | Blocked for VLMs — needs 2-line patch |
| Multi-GPU | FSDP2 works (minus OOM) | Single GPU only |

### CUDA Matching (CRITICAL)
- `causal-conv1d` compiles CUDA kernels at install time
- Must use `vastai/pytorch:2.10.0-cu128-cuda-12.9-mini-py312-2026-03-26` (toolkit 12.9, torch 12.8)
- NGC `26.01-py3` has toolkit 13.1 but torch 12.8 -> FAILS
- `causal-conv1d` setup.py HARDCODES all architectures — must clone, patch to SM 9.0 only, build from source
