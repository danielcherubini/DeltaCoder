# DeltaCoder v1.2 — Redo Plan

**Goal:** Redo v1.2 from pretokenize onwards, fixing the `normalize_messages()` bug that corrupted 65K+ tool-calling training examples and caused the model to produce degenerate thinking loops. Same dataset, same configs — better infrastructure strategy.

**Architecture:** Fix pretokenize.py → re-tokenize 230K dataset → SFT on Vast.ai H100 (Axolotl template) → validate SFT model → swap to PyTorch template for DPO pair gen (vLLM) + DPO training → merge + GGUF export on Romulus → test on Kronk → upload to HuggingFace.

**Tech Stack:** Axolotl (SFT), vLLM nightly (DPO pair gen), HuggingFace + PEFT + TRL (DPO training), ik_llama.cpp (GGUF export)

---

## What went wrong in the first attempt

The `normalize_messages()` function in `v1.2/scripts/pretokenize.py` (lines 36-65) converted structured `tool_calls` into raw `<functioncall>` text strings instead of passing them through to `apply_chat_template`. This corrupted ~65K+ function-calling training examples from xlam, coderforge, and other datasets. The SFT model trained on this corrupted data was degraded, the DPO pairs generated from it were also degraded, and the final model exhibited degenerate infinite thinking loops — worse than v1.1. Total wasted cost: ~$100.

## Infrastructure Strategy

### Vast.ai Shared Volume

Create a shared volume to persist data across instance swaps. This eliminates all the uploading/downloading between phases — the volume stays mounted and data persists even when instances are destroyed.

```bash
# Find a volume offer near cheap H100s (Europe preferred for latency)
vastai search volumes 'disk_space>=100 reliability>0.95' -o 'storage_cost'

# Create 100GB volume (enough for dataset + model + adapters)
vastai create volume <OFFER_ID> -s 100 -n "deltacoder-v1.2"
```

**Volume contents (persist across all phases):**
```
/workspace/volume/
├── data/
│   ├── v1.2_sft_train.jsonl          # 230K raw rows (3.6GB)
│   └── train_tokenized.jsonl         # Pre-tokenized output (~10GB)
├── scripts/                          # All training scripts
├── configs/                          # Axolotl config
├── merged_v1.2/                      # Merged SFT model (17GB)
├── dpo_pairs_v1.2.jsonl              # DPO pairs (~27MB)
├── sft_adapter/                      # SFT LoRA adapter (~1.3GB)
├── dpo_adapter/                      # DPO LoRA adapter (~350MB)
└── logs/                             # Training logs
```

### Instance Strategy: Swap Templates, Keep Volume

| Phase | Template | GPU | Why | Est. Cost |
|-------|----------|-----|-----|-----------|
| Pretokenize + SFT | **Axolotl** | 1x H100 SXM 80GB | Axolotl pre-installed, packing works. H100 has same compute as H200 — the extra VRAM is unused (LoRA only uses ~45GB). ~$1.47/hr. | ~$68 |
| SFT Validation + DPO pair gen + DPO training | **PyTorch** (CUDA 12.8) | 1x H100 SXM 80GB | vLLM nightly for serving, TRL for DPO. Need to `pip install` vLLM, transformers>=5.0, trl, peft, flash-linear-attention, causal-conv1d. ~$1.47/hr. | ~$12 |

**Why H100 not H200:** H200 was $2.58/hr vs H100 at $1.47/hr. The v1.2 SFT training used only ~45GB VRAM — H100's 80GB is more than enough. The compute (FLOPs) is identical. Saves ~40% on the biggest cost item (SFT training).

**Why swap templates:** Axolotl template has axolotl pre-installed (saves 30+ min setup), but DPO pair gen needs vLLM nightly which conflicts with Axolotl's pinned transformers 4.x. PyTorch template gives a clean venv for vLLM + transformers 5.x. The shared volume means swapping is instant — no data transfer.

### Task 1: Fix `pretokenize.py` `normalize_messages()` (LOCAL)

**Context:**
The `normalize_messages()` function currently flattens structured `tool_calls` into `<functioncall>` text strings. This is wrong — `apply_chat_template` knows how to handle structured tool calls natively (Qwen3.5's template renders them as XML `<tool_call>` blocks). We need to pass tool_calls through as-is and let `apply_chat_template` handle formatting.

**Files:**
- Modify: `v1.2/scripts/pretokenize.py`

**What to implement:**
Change `normalize_messages()` to preserve the `tool_calls` key on assistant messages instead of flattening them into the content string. The function should:
1. Keep `role` and `content` (as before)
2. If `tool_calls` is present on an assistant message, include it as-is in the output dict
3. If `tools` is present at the conversation level, pass it to `apply_chat_template` via the `tools` parameter
4. Keep the `content` string normalization (empty string fallback, type coercion)

Also update `tokenize_conversation()` to accept and pass `tools` to `apply_chat_template`.

Also update `process_chunk()` to pass `tools` from the row through to `tokenize_conversation()`.

**Steps:**

- [ ] Verify output has non-zero trainable tokens for tool-calling examples

**Phase 4b: Validate SFT Model (STOP/GO gate)**

Serve the merged SFT model via vLLM and test with 5+ prompts before spending money on DPO.

Test prompts (minimum):
1. Simple coding: "Write a Python function to check if a number is prime"
2. Multi-turn coding: "Write a linked list in Python" → "Add a reverse method"
3. Tool calling: Provide a function schema and ask the model to call it
4. Reasoning: "Explain the time complexity of quicksort" (should use `<think>` without looping)
5. Code review: "Review this code for bugs: [paste buggy code]"

**Failure criteria (STOP and investigate if any):**
- Model produces degenerate thinking loops (repeating same sentence)
- Model outputs `<functioncall>` text instead of structured tool calls
- Model produces empty or nonsensical responses
- Model fails to stop generating (no EOS)

**Phase 4c: DPO Pair Generation**

Same process as original v1.2 — generate on-policy pairs from AceCode-V2-122K using `generate_dpo_pairs.py`. Expected ~4hrs, ~4K-5K pairs.

**Phase 4d: DPO Training**

Run DPO training with `train_dpo.py`. Expected ~283 steps, ~2-4hrs.
