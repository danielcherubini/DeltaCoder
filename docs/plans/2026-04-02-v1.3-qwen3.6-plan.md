# DeltaCoder v1.3 — Qwen3.6 Base Model Plan

**Date:** 2026-04-02
**Status:** BLOCKED — waiting for Qwen3.6 open weights release
**Base Model:** Qwen/Qwen3.6-9B (name TBD)

## Overview

v1.3 pivots from Qwen3.5-9B to **Qwen3.6** as the base model. Qwen3.6-Plus was announced
with API access and open weights "in the next few days" per Chujie Zheng on X.

Key improvements in Qwen3.6:
- 1M context window
- Significantly improved agentic coding capability
- Better multimodal perception
- Beating GPT5.2 and Claude 4.5 Opus on several benchmarks
- "From frontend web development to complex, repository-level problem solving"

## Pipeline (same as v1.2 — all scripts adapted)

1. **Verify architecture** — When weights drop:
   - Check if still GDN (Gated Delta Net) or new attention mechanism
   - Check model sizes available (~9B target)
   - Verify tokenizer (same vocab? Need re-tokenization?)
   - Check if still unified VLM (ForConditionalGeneration)
   - Test if VLM packing patch still applies to new Unsloth version

2. **Data preparation** — Reuse v1.2 pruned dataset:
   - If tokenizer unchanged: reuse `v1.2/data/v1.2_pretokenized_pruned.parquet` directly
   - If tokenizer changed: re-tokenize from `v1.2/data/v1.2_sft_train_pruned.jsonl` (148,976 rows)
   - Run: `python v1.3/scripts/pretokenize_for_sft.py --data <jsonl> --output <parquet>`

3. **SFT Training** — LoRA at 32K context:
   - Same hyperparameters as v1.2 (validated across 4 dry runs)
   - LoRA r=64, alpha=32, all GDN + attention + MLP targets
   - batch_size=1, grad_accum=4, packing=True
   - Run: `python v1.3/scripts/train_unsloth.py --data /workspace/v1.3_pretokenized.parquet`

4. **Merge SFT** — Unsloth merge_and_unload

5. **DPO** — On-policy preference pairs:
   - Generate DPO pairs with `v1.3/scripts/generate_dpo_pairs.py`
   - Train with `v1.3/scripts/train_dpo.py`

6. **Export** — GGUF quantization + upload:
   - `v1.3/scripts/merge_and_export_dpo.py`
   - Test on Kronk before HF upload

## Training Strategy: Chunked Iteration

Instead of one monolithic ~177h run, we'll split training into chunks to enable
evaluation between each chunk. Two approaches considered:

### Option A: Chunked Shuffled (3x ~50K rows each)
Split the full 149K shuffled dataset into 3 equal chunks, all domains mixed.
Resume LoRA adapter between chunks.

- **Pro**: No catastrophic forgetting (domains mixed), can stop early if good enough
- **Pro**: Each chunk ~$70-87 on H100 — spot instance friendly
- **Con**: Can't isolate which dataset helps/hurts

### Option B: Sequential Domain Training
Train on one dataset source at a time (nemotron_tool_calling → opencoder_reasoning → etc).

| Run | Dataset | Rows | ~H100 Time | ~Cost |
|-----|---------|------|-----------|-------|
| 1 | nemotron_tool_calling | 40K | ~48h | ~$70 |
| 2 | opencoder_reasoning | 25K | ~30h | ~$44 |
| 3 | swesmith | 20K | ~24h | ~$35 |
| 4 | nemotron_agentic | 19K | ~23h | ~$33 |
| 5 | code_feedback | 15K | ~18h | ~$26 |
| 6 | xlam | 15K | ~18h | ~$26 |
| 7 | nemotron_swe | 10K | ~12h | ~$18 |
| 8 | magicoder | 5K | ~6h | ~$9 |

- **Pro**: Can eval after each domain, see exactly what each dataset contributes
- **Pro**: Can stop, reorder, skip, or repeat domains based on eval
- **Pro**: If a dataset hurts performance, just roll back that adapter
- **Con**: Catastrophic forgetting risk (later domains overwrite earlier ones)
- **Con**: Order-dependent, LR scheduling is awkward
- **Mitigation**: LoRA helps (base weights frozen), domains overlap, can use replay

### Recommended: Hybrid Approach
1. Run 2-3 cheap single-domain ablations on questionable datasets
2. Based on results, adjust mix and split into 3 shuffled chunks
3. Train chunk by chunk, eval between each, stop when good enough
4. Total cost: ~$100 ablations + $87-261 for 1-3 chunks = $187-361

Technical requirements for chunked training:
- `train_unsloth.py` needs `--resume-from <adapter_path>` support
- LR scheduler: warmup on chunk 1, no warmup on subsequent chunks
- Data: `split_dataset.py` to partition pretokenized parquet into N chunks
- Each chunk must be a random sample (not sorted by source)

## Evaluation Plan

### Existing Baselines

| Model | HumanEval | HumanEval+ | Terminal-Bench Easy |
|-------|-----------|------------|---------------------|
| Qwen3.5-9B (base) | 53.7% | — | — |
| DeltaCoder-9B v1 (temp=0.6) | 50.6% | 49.4% | 2/4 (50%) |
| DeltaCoder-9B v1.1-DPO (temp=0.6) | TBD | TBD | 2/4 (50%)* |

**Key finding**: v1 REGRESSED on HumanEval vs base model (50.6% vs 53.7%).
v1.3 must beat the base model on coding while adding tool calling + agentic.

### Eval Tiers

**Tier 1 — Between every chunk (~15 min):**
- EvalPlus HumanEval+ pass@1 (temp=0.6 to match existing baselines)
- 164 problems, test-case verified, runs on training GPU via vLLM
- `pip install evalplus` — one command
- **Goal**: Beat base model (>53.7% HumanEval), don't regress

**Tier 2 — After major milestones (~45 min):**
- BFCL (Berkeley Function Calling Leaderboard) — tool calling accuracy
- LiveCodeBench or BigCodeBench — harder/contamination-free coding
- `pip install bfcl-eval` — packaged on PyPI
- **Goal**: Establish tool calling baseline on Qwen3.6, then beat it

**Tier 3 — Before shipping:**
- GGUF it, deploy to Kronk, real-world testing
- Optionally Terminal-Bench 2.0 on easy subset (but it's slow — hours per task)

### Eval Workflow
1. When Qwen3.6 weights drop: run Tier 1 + Tier 2 on base model → establish baselines
2. After each training chunk: merge LoRA → serve with vLLM → run Tier 1 → compare
3. After full SFT: run Tier 2
4. After DPO: run Tier 2 + Tier 3 (Kronk)
5. Ship only if ALL metrics beat the base model

### Benchmarks NOT Used (and why)
- **AIME 2025**: Math competition — not our priority
- **GPQA Diamond**: Graduate-level science — not our priority
- **Terminal-Bench 2.0 (full)**: 89 Docker tasks, hours per task, requires agent scaffold — too slow for iteration
- **SWE-bench**: Docker orchestration, hours of execution — too heavyweight
- **LM-Eval (MMLU, GSM8K)**: General capability, doesn't test our priorities
