# DeltaCoder Qwen3.6 v1.0 — Qwen3.6 Base Model Plan

**Date:** 2026-04-02
**Status:** BLOCKED — waiting for Qwen3.6 open weights release
**Base Model:** Qwen/Qwen3.6-9B (name TBD)

## Overview

Qwen3.6 v1.0 pivots from Qwen3.5-9B to **Qwen3.6** as the base model. Qwen3.6-Plus was announced
with API access and open weights "in the next few days" per Chujie Zheng on X.

Key improvements in Qwen3.6:
- 1M context window
- Significantly improved agentic coding capability
- Better multimodal perception
- Beating GPT5.2 and Claude 4.5 Opus on several benchmarks
- "From frontend web development to complex, repository-level problem solving"

## Pipeline (same as qwen3.5/v1.1 — all scripts adapted)

1. **Verify architecture** — When weights drop:
   - Check if still GDN (Gated Delta Net) or new attention mechanism
   - Check model sizes available (~9B target)
   - Verify tokenizer (same vocab? Need re-tokenization?)
   - Check if still unified VLM (ForConditionalGeneration)
   - Test if VLM packing patch still applies to new Unsloth version

2. **Data preparation** — Build revised Jackrong-inspired dataset (~157K rows, ~700M tokens):
    - Download new sources: `preprocess_competitive_programming.py`, `preprocess_qwen3_coder_distill.py`
    - Apply tiered token filtering: `filter_for_v12_pruned.py` (8K Tier 1, 16K Tier 2)
    - Build mix: `build_training_mix.py`
    - Pretokenize: `pretokenize_for_sft.py --data <jsonl> --output <parquet>`
    - If Qwen3.6 tokenizer unchanged from 3.5: can reuse existing tokenized data for old sources

3. **SFT Training** — LoRA at 32K context:
    - LoRA r=64, **alpha=64** (1:1 ratio — Jackrong-validated), all GDN + attention + MLP targets
    - batch_size=1, grad_accum=4, packing=True, **train_on_responses_only=True**
    - 1 epoch (157K rows, ~700M tokens — no need for 2 epochs)
    - Run: `python qwen3.6/v1.0/scripts/train_unsloth.py --data /workspace/v1.0_pretokenized.parquet`

4. **Merge SFT** — Unsloth merge_and_unload

5. **DPO** — On-policy preference pairs:
    - Generate DPO pairs with `qwen3.6/v1.0/scripts/generate_dpo_pairs.py`
    - Train with `qwen3.6/v1.0/scripts/train_dpo.py`

6. **Export** — GGUF quantization + upload:
    - `qwen3.6/v1.0/scripts/merge_and_export_dpo.py`
   - Test on Kronk before HF upload

## Training Strategy

Single monolithic run — dataset is ~$100-130, simple and clean:
- **1 epoch** — 157K rows at ~700M tokens, no repeat passes needed
- **`train_on_responses_only=True`** — mask user/system tokens, only train on assistant responses
- **`lora_alpha=64`** — 1:1 ratio with r=64, stronger LoRA influence
- All data mixed and shuffled (not sequential by domain)
- Eval after training, not during (training is now ~4 days not 7+)

## Evaluation Plan

### Existing Baselines

| Model | HumanEval | HumanEval+ | Terminal-Bench Easy |
|-------|-----------|------------|---------------------|
| Qwen3.5-9B (base) | 53.7% | — | — |
| DeltaCoder-9B v1 (temp=0.6) | 50.6% | 49.4% | 2/4 (50%) |
| DeltaCoder-9B v1.1-DPO (temp=0.6) | TBD | TBD | 2/4 (50%)* |

**Key finding**: v1 REGRESSED on HumanEval vs base model (50.6% vs 53.7%).
DeltaCoder Qwen3.6 v1.0 must beat the base model on coding while adding tool calling + agentic.

### Eval Tiers

**Tier 1 — Between every chunk (~15 min):**
- EvalPlus HumanEval+ pass@1 (temp=0.6 to match existing baselines)
- 164 problems, test-case verified, runs on training GPU via vLLM
- `pip install evalplus` — one command
- **Goal**: Beat base model (>53.7% HumanEval), don't regress

**Tier 2 — After major milestones (~45 min):**
- BFCL (Berkeley Function Calling Leaderboard) — tool calling accuracy
- LiveCodeBench or BigCodeBench — harder/contamination-free coding
- `pip install bfcl-eval` — packaged on PyPI
- **Goal**: Establish tool calling baseline on Qwen3.6, then beat it

**Tier 3 — Before shipping:**
- GGUF it, deploy to Kronk, real-world testing
- Optionally Terminal-Bench 2.0 on easy subset (but it's slow — hours per task)

### Eval Workflow
1. When Qwen3.6 weights drop: run Tier 1 + Tier 2 on base model → establish baselines
2. After full SFT: merge LoRA → serve with vLLM → run Tier 1 + Tier 2 → compare vs base
3. After DPO: run Tier 2 + Tier 3 (Kronk)
4. Ship only if ALL metrics beat the base model

### Benchmarks NOT Used (and why)
- **AIME 2025**: Math competition — not our priority
- **GPQA Diamond**: Graduate-level science — not our priority
- **Terminal-Bench 2.0 (full)**: 89 Docker tasks, hours per task, requires agent scaffold — too slow for iteration
- **SWE-bench**: Docker orchestration, hours of execution — too heavyweight
- **LM-Eval (MMLU, GSM8K)**: General capability, doesn't test our priorities

## Notes

- CoderForge intentionally excluded (teaches wrong tool-calling behavior)
- Must preserve vision capabilities (finetune_vision_layers=False)
- DO NOT use Unsloth DPOTrainer — crashes with VLM models
- DO NOT use flash_attention_2 with GDN — use SDPA instead
- `train_on_responses_only` uses Qwen chat template markers: instruction_part=`<|im_start|>user\n`, response_part=`<|im_start|>assistant\n`
- `nemotron_swe` dropped — 100% exceed 16K tokens, zero useful signal at any context limit
- `lora_alpha=64` (1:1 with r=64) — proven by Jackrong's Qwopus3.5 (+4.87pp HumanEval)
