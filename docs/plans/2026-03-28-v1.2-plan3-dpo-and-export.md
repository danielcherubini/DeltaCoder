# DeltaCoder v1.2 — Plan 3: DPO Training + GGUF Export

**Goal:** Generate on-policy DPO preference pairs from the v1.2 SFT model, run DPO training, merge both LoRA adapters, export all GGUF quants, and upload to HuggingFace.

**Prerequisites:**
- Plan 2 completed — v1.2 SFT LoRA adapter exists at `outputs/deltacoder-9b-v1.2/lora_adapter/`
- The v1.1 DPO pipeline scripts exist and work: `scripts/generate_dpo_pairs.py`, `scripts/train_dpo.py`, `scripts/merge_and_export_dpo.py`

**Architecture:** Same pipeline as v1.1 — merge SFT LoRA → serve via vLLM → generate pairs → DPO train (HF+PEFT+TRL) → two-stage merge → GGUF export.

**Tech Stack:** vLLM (inference), HuggingFace + PEFT + TRL (DPO training, NOT Unsloth), llama.cpp (GGUF export)

⚠️ **DO NOT use Unsloth DPOTrainer.** It has a known bug where it detects Qwen3.5 as a vision model and crashes with `KeyError: 'images'`. Use `scripts/train_dpo.py` which uses plain HuggingFace + PEFT + TRL.

### Task 1: Merge SFT LoRA + Serve with vLLM for Pair Generation

**Context:**
The DPO pair generation script (`scripts/generate_dpo_pairs.py`) needs an OpenAI-compatible API endpoint serving the v1.2 SFT model. We merge the SFT LoRA into the base model, save full bf16 weights, then serve with vLLM. This must be done on a cloud GPU instance since the merged model needs ~18GB VRAM.

Qwen3.5-9B is technically a VLM so vLLM may auto-detect it as multimodal. If it crashes during loading, add `--trust-remote-code`. If it shows `<think>` blocks in output, add `--chat-template-kwargs '{"enable_thinking":false}'` to the request body (the script already does this via `chat_template_kwargs` in the request).

**Files:**
- No script changes needed — reuse existing `scripts/generate_dpo_pairs.py`

**What to do:**

**Step 1: Rent Vast.ai instance**
- Template: vLLM (not Unsloth — this step is inference, not training)
- GPU: H100 SXM 80GB (or cheaper H100 NVL — just need fast inference)
- Disk: ≥100GB

**Step 2: Setup**
```bash
TERM=xterm-256color ssh -p <PORT> root@<IP>
git clone https://github.com/danielcherubini/DeltaCoder.git /workspace/DeltaCoder
mkdir -p /workspace/DeltaCoder/data /workspace/DeltaCoder/logs /workspace/merged_v1.2
pip install peft transformers aiohttp openai numpy datasets tqdm -q
```

**Step 3: Upload SFT adapter** (from romulus)
```bash
rsync -avP ~/Coding/AI/DeltaCoder/outputs/deltacoder-9b-v1.2/lora_adapter/ \
    -e "ssh -p <PORT>" root@<IP>:/workspace/DeltaCoder/outputs/deltacoder-9b-v1.2/lora_adapter/
```

**Step 4: Merge SFT LoRA into base** (on instance)
```bash
python3 -c "
from peft import PeftModel
from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

print('Loading base model...')
base = AutoModelForCausalLM.from_pretrained(
    'Qwen/Qwen3.5-9B',
    torch_dtype=torch.bfloat16,
    device_map='cpu',
    trust_remote_code=True,
    attn_implementation='eager',
)
print('Loading SFT adapter...')
model = PeftModel.from_pretrained(base, '/workspace/DeltaCoder/outputs/deltacoder-9b-v1.2/lora_adapter')
print('Merging...')
model = model.merge_and_unload()
print('Saving...')
model.save_pretrained('/workspace/merged_v1.2')
AutoTokenizer.from_pretrained('Qwen/Qwen3.5-9B', trust_remote_code=True).save_pretrained('/workspace/merged_v1.2')
print('Done')
"
```

**Step 5: Serve with vLLM**
```bash
nohup python -m vllm.entrypoints.openai.api_server \
    --model /workspace/merged_v1.2 \
    --served-model-name deltacoder \
    --port 18000 \
    --host 0.0.0.0 \
    --dtype bfloat16 \
    --max-model-len 4096 \
    --enforce-eager \
    --enable-prefix-caching \
    > /workspace/logs/vllm.log 2>&1 &

# Wait for server
sleep 60 && curl -s http://localhost:18000/health
```

**Step 6: Generate DPO pairs**
```bash
cd /workspace/DeltaCoder && nohup env HF_TOKEN=$HF_TOKEN python scripts/generate_dpo_pairs.py \
    --n-problems 10000 \
    --n-samples 8 \
    --api-base "http://localhost:18000/v1" \
    --model deltacoder \
    --output data/dpo_pairs_v1.2.jsonl \
    --concurrency 32 \
    > logs/generate_dpo_v1.2.log 2>&1 &
```

Monitor: `tail -f /workspace/DeltaCoder/logs/generate_dpo_v1.2.log`

Expected: ~4hrs, ~4500 pairs at 45% keep rate.

**Step 7: Download pairs** (from romulus)
```bash

    root@<IP>:/workspace/DeltaCoder/data/dpo_pairs_v1.2.jsonl \
    ~/Coding/AI/DeltaCoder/data/
```

### Task 3: Two-Stage Merge + GGUF Export + Upload

**Context:**
Merge both LoRA adapters (SFT + DPO) sequentially into the base model, export all GGUF quants, and upload to HuggingFace. The existing `scripts/merge_and_export_dpo.py` handles this — just update the default paths and GGUF filename pattern.

The merge sequence is:
```
Qwen/Qwen3.5-9B (base)
  + v1.2 SFT LoRA → merge → merged_v1.2
  + v1.2 DPO LoRA → merge → merged_v1.2_final
  → convert to f16 GGUF
  → quantize all quants (each uploaded + deleted to save disk)
```

**Files:**
- Modify: `scripts/merge_and_export_dpo.py` (update default paths and filename pattern)

**What to implement:**

Update these defaults in `scripts/merge_and_export_dpo.py`:

```python
# Update default args in parse_args():
parser.add_argument("--sft-adapter", default="./outputs/deltacoder-9b-v1.2/lora_adapter")
parser.add_argument("--dpo-adapter", default="./outputs/deltacoder-9b-v1.2-dpo/lora_adapter")
parser.add_argument("--merged-dir", default="./outputs/deltacoder-9b-v1.2-merged")
parser.add_argument("--gguf-dir", default="./outputs/deltacoder-9b-v1.2-gguf")
```

Update GGUF filename pattern — add a constant and use it everywhere:
```python
FILENAME_PREFIX = "DeltaCoder-9B-v1.2"
```
Then replace all hardcoded `DeltaCoder-9B-v1.1-DPO` with `{FILENAME_PREFIX}`:
- `f16_gguf = f"{gguf_dir}/{FILENAME_PREFIX}-f16.gguf"`
- BF16 copy: `f"{gguf_dir}/{FILENAME_PREFIX}-BF16.gguf"`
- Quant loop: `f"{gguf_dir}/{FILENAME_PREFIX}-{quant}.gguf"`

Also restore the full QUANTS list (the current list has 6 quants removed from a previous run — they must all be generated for v1.2):
