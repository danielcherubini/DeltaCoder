## What Changed From Original Plan

### Framework: Unsloth → Axolotl

The original plan used **Unsloth** with `FastVisionModel`. After extensive testing, we switched to **Axolotl** because **Unsloth has no sample packing for VLMs** (Qwen3.5 is classified as a VLM).

| Approach | Steps | Per-step | Total | Cost |
|----------|-------|----------|-------|------|
| Unsloth bs=2, H100 | 13,408 | 25.5s | 95hrs | $150 |
| Unsloth bs=4, H100 | 7,188 | 68s | 132hrs | $209 |
| Unsloth bs=8, H100 | — | OOM | — | — |
| Unsloth bs=4, H200 141GB | 7,188 | 66s | 132hrs | $264 |
| Unsloth bs=8, H200 141GB | — | OOM | — | — |
| **Axolotl + packing, H200** | **~23K** | **~7s** | **~46hrs** | **~$87** |

### Qwen3.5 GDN Architecture Limitations (discovered during training)

**Gated Delta Net (GDN) does not support `micro_batch_size > 1` with sample packing.** This is a known limitation confirmed by:
- [NVIDIA/Megatron-LM #3798](https://github.com/nvidia/megatron-lm/issues/3798) — open issue requesting packed sequence support for GDN
- [axolotl-ai-cloud/axolotl #3453](https://github.com/axolotl-ai-cloud/axolotl/issues/3453) — sample packing causes loss=0 with Qwen3.5
- [ms-swift Qwen3.5 Best Practices](https://swift.readthedocs.io/en/latest/BestPractices/Qwen3_5-Best-Practice.html) — official docs confirm "transformers' GatedDeltaNet does not support packing/padding_free"

The GDN layers use `cu_seqlens` (cumulative sequence lengths) which requires batch dimension = 1 when packing multiple sequences into a single window. This applies to ALL attention backends (SDPA, flash_attention_2).

**flash_attention_2 + Qwen3.5 GDN = CUDA illegal memory access.** Even at batch=1, FA2 causes `cudaErrorIllegalAddress` in the GDN monkeypatch. Must use SDPA.

**Workaround from ms-swift:** Use `group_by_length: true` to group similar-length sequences together, reducing padding waste within micro-batches.

### Failed approaches (for reference)
1. Unsloth bs=2 on H100 80GB: 95hrs — too slow, no packing
2. Unsloth bs=4 on H100 80GB: 132hrs — too slow
3. Unsloth bs=8 on H100 80GB: OOM
4. Unsloth bs=8 on H200 141GB: OOM (even 141GB not enough at 4096 seq)
5. Unsloth bs=4 on H200 141GB: 132hrs — same speed as H100, compute-bound
6. Cutting seq_length to 2048: quality tradeoff on CoderForge trajectories
7. Dropping OpenCodeReasoning: lost reasoning capability
8. Axolotl + FA2 + batch=4: `cu_seqlens` requires batch=1 (GDN limitation)
9. Axolotl + FA2 + batch=1: CUDA illegal memory access in GDN layers
10. Installing flash-attn from source: 30+ min compile, didn't help due to #8/#9

### Vast.ai instance management
- **Instance 1:** H100 SXM 80GB (Unsloth template) — killed, too slow
- **Instance 2:** H200 141GB (Unsloth template) — killed, same speed + flash-attn build issues, 84GB of HF dataset cache filled 100GB disk
- **Instance 3:** H200 141GB (Axolotl template, France, $1.9/hr) — current, pre-installed axolotl 0.16 + flash-attn 2.8.3

**Lesson learned:** Use Vast.ai shared drives to persist datasets across instance swaps. Use the Axolotl template from the start for Qwen3.5.

## Execution Steps

### Setup (on Vast.ai H200 — Axolotl template)
```bash
# Axolotl template already has: axolotl, flash-attn, transformers, peft, accelerate

# Set HF token (Vast.ai scrubs inline env vars — must export)
export HF_TOKEN=<token>

# Create dirs and upload files
mkdir -p /workspace/DeltaCoder/{scripts,data,logs,outputs,configs}
# rsync pretokenize.py, config yaml, and v1.2_sft_train.jsonl from local

# Pre-tokenize (~5 mins)
python scripts/pretokenize.py data/v1.2_sft_train.jsonl /dev/shm/train_tokenized.jsonl

# Train (nohup so it survives SSH disconnect)
cd /workspace/DeltaCoder
nohup accelerate launch -m axolotl.cli.train configs/deltacoder-9b-lora-v1.2.yaml \
    > logs/v1.2_axolotl_train.log 2>&1 &
tail -f logs/v1.2_axolotl_train.log
```

### Disk space management
The Axolotl template pre-installs ~30GB of packages. With 100GB disk:
- `/venv` or conda env: ~15-30GB
- HF model cache: ~18GB (Qwen3.5-9B)
- Dataset: ~3.6GB
- Pre-tokenized: ~10GB (on /dev/shm, not disk)
- Checkpoints: ~1.3GB per save

**Watch out for:** HF dataset cache (`~/.cache/huggingface/datasets` or `/workspace/.hf_home/datasets`) can grow to 60GB+. Clean with `rm -rf` if disk fills up.

### After training

#### 0. Clean HF datasets cache (free ~83GB before merge)
```bash
rm -rf /workspace/data/huggingface-cache/datasets
# /workspace/data/huggingface-cache/hub (18GB, Qwen3.5-9B weights) — KEEP
```

#### 1. Download adapter locally
```bash
rsync -avP -e "ssh -p <PORT>" root@<IP>:/workspace/DeltaCoder/outputs/deltacoder-9b-v1.2/ \
    ~/Coding/AI/DeltaCoder/outputs/deltacoder-9b-v1.2/
```

#### 2. Merge SFT LoRA into base model (on same H200)
```bash
python3 -c "
from peft import PeftModel
from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

print('Loading base model...')
base = AutoModelForCausalLM.from_pretrained(
    'Qwen/Qwen3.5-9B',
    torch_dtype=torch.bfloat16,
    device_map='cpu',
    trust_remote_code=True,
    attn_implementation='eager',
)
print('Loading SFT adapter...')
model = PeftModel.from_pretrained(base, './outputs/deltacoder-9b-v1.2')
print('Merging...')
model = model.merge_and_unload()
print('Saving...')
model.save_pretrained('/workspace/merged_v1.2')
AutoTokenizer.from_pretrained('Qwen/Qwen3.5-9B', trust_remote_code=True).save_pretrained('/workspace/merged_v1.2')
print('Done')
"
```
