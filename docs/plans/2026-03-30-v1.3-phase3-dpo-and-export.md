**Status:** ⏳ PENDING — waiting for Phase 2 (SFT) to complete

**Goal:** Generate on-policy DPO preference pairs from the v1.3 SFT model, run DPO training, merge both LoRA adapters, export all GGUF quants, upload to HuggingFace, and run Terminal-Bench 2 evaluation.

**Prerequisites:**
- Phase 2 complete — `danielcherubini/Qwen3.5-DeltaCoder-9B-v1.3-SFT` uploaded to HF
- Merged SFT model at `/workspace/merged_v1.3` on H200

- At least 60GB disk free on `/workspace`

**Architecture:** Merge SFT LoRA → serve via vLLM → generate pairs → DPO train → two-stage merge → GGUF export.

**Tech Stack:** vLLM (inference), HuggingFace + PEFT + TRL (DPO, NOT Unsloth), llama.cpp (GGUF export), hf-mount (optional lazy model loading)

⚠️ **DO NOT use Unsloth DPOTrainer.** It detects Qwen3.5 as a VLM and crashes with `KeyError: 'images'`. Use `scripts/train_dpo.py` which uses plain HF + PEFT + TRL.

⚠️ **DO NOT use flash_attention_2.** Causes `cudaErrorIllegalAddress` with Qwen3.5 GDN layers. Use SDPA.

## Task 1: Generate DPO Pairs

**Context:**
Generate on-policy DPO preference pairs from the v1.3 SFT model. The script `scripts/generate_dpo_pairs.py` sends coding problems to the model, samples 8 responses per problem, and keeps pairs where at least 1 passes and at least 1 fails execution verification. This produces high-quality preference data anchored to actual code correctness.

The merged SFT model already exists at `/workspace/merged_v1.3`. Use that directly rather than mounting from HF.

If `/workspace/merged_v1.3` is unavailable for any reason, mount from HF instead:
```bash
mkdir -p /mnt/sft-model
hf-mount start --hf-token $HF_TOKEN \
    repo danielcherubini/Qwen3.5-DeltaCoder-9B-v1.3-SFT /mnt/sft-model
# Then use /mnt/sft-model as --model below
```

**Step 1: Serve merged SFT model with vLLM**
```bash
nohup python -m vllm.entrypoints.openai.api_server \
    --model /workspace/merged_v1.3 \
    --served-model-name deltacoder \
    --port 18000 \
    --host 0.0.0.0 \
    --dtype bfloat16 \
    --max-model-len 4096 \
    --enforce-eager \
    --enable-prefix-caching \
    > logs/vllm_v1.3.log 2>&1 &

# Wait for server ready
sleep 60 && curl -s http://localhost:18000/health
```

**Step 2: Generate pairs**
```bash
cd /workspace/DeltaCoder
nohup python scripts/generate_dpo_pairs.py \
    --n-problems 10000 \
    --n-samples 8 \
    --api-base "http://localhost:18000/v1" \
    --model deltacoder \
    --output data/dpo_pairs_v1.3.jsonl \
    --concurrency 32 \
    > logs/generate_dpo_v1.3.log 2>&1 &

tail -f logs/generate_dpo_v1.3.log
```

Expected: ~4hrs, ~4,000-5,000 pairs at 40-50% keep rate.

**Step 3: Download pairs locally**

## Task 4: Terminal-Bench 2 Evaluation

**Context:**
Terminal-Bench 2 has 87 tasks. With vLLM serving the merged DPO model on the H200 (141GB VRAM, ~120GB free after loading 9B model), all 87 tasks can run in parallel using Docker containers. Total eval time ≈ time for the slowest single task (~5-10 min) rather than sequential (7-14hrs).

This produces a full benchmark score for the model card while the instance is still running. Cost: ~$0.50 extra. Do this BEFORE tearing down the instance.

**Note on llama.cpp vs ik_llama:** llama.cpp has a known GPU offload bug with Qwen3.5 hybrid GDN architecture — it doesn't fully offload GDN layers to GPU, running at ~110 tok/s instead of full speed. Previous Terminal-Bench results used llama.cpp on remote Vast.ai instances, which may have caused artificial timeouts. v1.3 eval uses vLLM (native BF16, full GPU utilization) — results will be directly comparable to properly accelerated inference.

### Option A: Full parallel eval on H200 (preferred — do before teardown)

**Step 1: Serve DPO-merged model with vLLM**

Use `./outputs/deltacoder-9b-v1.3-merged` — the DPO-merged model kept by `--keep-merged` in Task 3. This is the final model (SFT + DPO merged), not the SFT-only model at `/workspace/merged_v1.3`.

```bash
cd /workspace/DeltaCoder
nohup python -m vllm.entrypoints.openai.api_server \
    --model ./outputs/deltacoder-9b-v1.3-merged \
    --served-model-name deltacoder \
    --port 18000 \
    --host 0.0.0.0 \
    --dtype bfloat16 \
    --max-model-len 8192 \
    --gpu-memory-utilization 0.85 \
    > logs/vllm_eval.log 2>&1 &
sleep 60 && curl -s http://localhost:18000/health
```

**Step 2: Run all 87 Terminal-Bench tasks in parallel**

Terminal-Bench 2 supports `--n-concurrent`. With 87 tasks and Docker available, run all at once:
```bash
# Adjust path/flags to match your Terminal-Bench 2 setup
OPENAI_API_KEY=sk-none python -m terminalbench.run \
    --api-base http://localhost:18000/v1 \
    --model deltacoder \
    --n-concurrent 87 \
    --timeout-multiplier 2 \
    --output-dir logs/terminalbench_v1.3
```

Expected: ~10-15 minutes for all 87 tasks.

**Step 3: Kill vLLM, delete merged model, tear down instance**
```bash
pkill -f vllm

# Delete the DPO-merged model (was kept for eval, no longer needed)
rm -rf /workspace/DeltaCoder/outputs/deltacoder-9b-v1.3-merged

# Now safe to destroy the Vast.ai instance
```
