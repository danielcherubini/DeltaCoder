# DeltaCoder v1.2 — Phase 3: DPO Training + GGUF Export

**Status:** ⏳ PENDING — Phase 2 finishes ~5:20 PM CEST 2026-03-31, then start immediately

**Goal:** Generate on-policy DPO preference pairs from the v1.2 SFT model, run DPO training, merge both LoRA adapters, export all GGUF quants, and upload to HuggingFace.

**Prerequisites:**
- Phase 2 completed — v1.2 SFT LoRA adapter merged + uploaded to HF
- Same H200 instance from Phase 2 (TRL, PEFT already installed — just need vLLM)

**Architecture:** Merge SFT LoRA → serve via vLLM → generate pairs → DPO train (HF+PEFT+TRL) → two-stage merge → GGUF export.

**Tech Stack:** vLLM nightly (inference — stable releases fail, must use nightly), HuggingFace + PEFT + TRL (DPO training, NOT Unsloth), llama.cpp (GGUF export)

⚠️ **DO NOT use Unsloth DPOTrainer.** It has a known bug where it detects Qwen3.5 as a vision model and crashes with `KeyError: 'images'`. Use `scripts/train_dpo.py` which uses plain HuggingFace + PEFT + TRL.

⚠️ **vLLM requires special config wrapping for Qwen3.5-9B text-only.** The merged SFT model has `model_type: qwen3_5_text` which vLLM doesn't natively support. Must wrap config.json using the official Qwen3.5-9B VL config as template, use a separate venv with `transformers>=5.0`, and serve with `--language-model-only`. See AGENTS.md for full setup.

### Task 1: Serve Merged SFT Model with vLLM for Pair Generation

**Context:**
The DPO pair generation script needs an OpenAI-compatible API endpoint serving the v1.2 SFT model. Use vLLM nightly in a separate venv with transformers 5.x and wrapped config (see AGENTS.md for details).

**On H200 instance:**

**Step 1: Fix config for vLLM compatibility (one-time)**

The merged SFT model has flat `qwen3_5_text` config — must wrap in VL format:
```python
import json
from huggingface_hub import hf_hub_download

path = hf_hub_download('Qwen/Qwen3.5-9B', 'config.json')
with open(path) as f:
    official = json.load(f)
with open('/workspace/merged_v1.2/config.json') as f:
    text_config = json.load(f)
official['text_config'] = text_config
with open('/workspace/merged_v1.2/config.json', 'w') as f:
    json.dump(official, f, indent=2)
```

Also remove `"tokenizer_class": "TokenizersBackend"` from `tokenizer_config.json` if present (axolotl artifact).

**Step 2: Serve with vLLM**
```bash
source /workspace/vllm-env/bin/activate
nohup vllm serve /workspace/merged_v1.2 \
    --port 18000 --host 0.0.0.0 \
    --max-model-len 4096 \
    --gpu-memory-utilization 0.90 \
    --enable-prefix-caching \
    --reasoning-parser qwen3 \
    --dtype auto \
    --language-model-only \
    > /workspace/vllm.log 2>&1 &

# First run JIT-compiles FlashInfer GDN kernels (~15min), then:
curl -s http://localhost:18000/health
```

**Step 3: Generate DPO pairs**
```bash
cd /workspace/DeltaCoder && nohup python scripts/generate_dpo_pairs.py \
    --n-problems 10000 \
    --n-samples 8 \
    --api-base "http://localhost:18000/v1" \
    --model /workspace/merged_v1.2 \
    --output data/dpo_pairs_v1.2.jsonl \
    --concurrency 32 \
    > logs/generate_dpo_v1.2.log 2>&1 &
```

Expected: ~4hrs, ~4500 pairs at 45% keep rate.

**Step 4: Download pairs locally**
```bash
scp -P <PORT> root@<IP>:/workspace/DeltaCoder/data/dpo_pairs_v1.2.jsonl ~/Coding/AI/DeltaCoder/v1.2/data/
```

**Acceptance criteria:**
- [ ] Merged model served via ik_llama-server (health check passes)
- [ ] Pair generation completes 10K problems
- [ ] ~4000+ preference pairs in `data/dpo_pairs_v1.2.jsonl`

## Post-Release: Evaluation

Two options depending on budget:

### Option A: Full parallel eval on H200 (before teardown — budget permitting)

Terminal-Bench 2 has 87 tasks. On the H200 with vLLM still running, all 87 can run simultaneously:
- vLLM serving DPO model: ~18-20GB VRAM, ~120GB free for KV cache
- 87 Docker containers in parallel — each task is independent
- Total time ≈ slowest single task (~5-10 min) instead of 7-14hrs sequential

This gives a full benchmark score for the model card at minimal extra cost (~$0.50 on top of Phase 3).

**Concurrency note:** `--n-concurrent` controls how many tasks run in parallel, but each task makes LLM calls that queue at the inference endpoint. With vLLM on H200, requests are batched natively so high concurrency actually helps — push `--n-concurrent 8` or higher. With ollama on romulus, only one request is processed at a time so concurrency >1 just stalls tasks waiting for the model — keep it at 1.

**Eval is now run from romulus WSL2** (`~/run-terminalbench.sh`), not from local Linux or Windows. Use `API_BASE=http://<H200-IP>:18000/v1` to point at H200 vLLM.
