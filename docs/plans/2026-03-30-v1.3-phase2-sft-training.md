## Hardware Strategy

### Primary: Multi-GPU DDP (recommended)

**OmniCoder (Tesslate) confirmed that 4x H200 DDP works with Qwen3.5-9B + Axolotl + LoRA + sample packing.** They trained 425K rows this way with 99.35% packing efficiency. DDP (not FSDP) means each GPU holds the full model — 9B in BF16 ≈ 18GB fits easily, leaving the rest for activations/gradients/packing.

With DDP on N GPUs, each GPU processes `1/N` of the data in parallel. Step time stays the same but you do N steps worth of data per wall-clock step. This directly divides training time by N.

**Search for instances:**
```bash
# 4x H200 (ideal — proven by OmniCoder)
vastai search offers 'gpu_name=H200 num_gpus=4 dph<10.0 reliability>0.95' -o 'dph'

# 2x H200 (cheaper, still 2x speedup)
vastai search offers 'gpu_name=H200 num_gpus=2 dph<5.0 reliability>0.95' -o 'dph'

# Alternative: 4x A6000 48GB (192GB total, cheaper)
vastai search offers 'gpu_name=RTX_A6000 num_gpus=4 dph<3.0 reliability>0.95' -o 'dph'
```

| Setup | VRAM/GPU | Total VRAM | Est $/hr | Training time | Est cost |
|---|---|---|---|---|---|
| **4x H200 DDP** | 141GB | 564GB | ~$8 | ~13-15hrs | ~$105-120 |
| **2x H200 DDP** | 141GB | 282GB | ~$4 | ~25-28hrs | ~$100-112 |
| **4x A6000 DDP** | 48GB | 192GB | ~$1.50 | ~13-15hrs | ~$20-23 |
| 1x H200 (fallback) | 141GB | 141GB | ~$2 | ~50-55hrs | ~$100-110 |

⚠️ **4x A6000 (48GB each):** Model fits (18GB) but 32768 seq_len activations may OOM on 48GB. Dry run required.

⚠️ **DDP requires `gradient_accumulation_steps` adjustment.** With 4 GPUs, effective batch = `micro_batch_size × ga × num_gpus`. To maintain the same effective batch of 4: set `gradient_accumulation_steps: 1` on 4 GPUs (was 4 on 1 GPU).

### Fallback: Single H200

If multi-GPU DDP fails or no multi-GPU instance is available, fall back to 1x H200. Same proven setup as v1.2, just with `seq_len: 32768`.

### Step 0: Multi-GPU dry run (before committing to full training)

Before renting a multi-GPU instance for the full run, do a quick validation:

1. Rent the cheapest multi-GPU option available
2. Sync `v1.3/` folder, pretokenize
3. Run 20-step dry run with DDP
4. If it works: commit to the full run
5. If it crashes: tear down immediately (~$2 lost), fall back to 1x H200

**DDP config addition for `v1.3/configs/axolotl.yaml` (only when using multi-GPU):**
```yaml
# Add for multi-GPU DDP — remove or set to 4 for single GPU
gradient_accumulation_steps: 1  # 4 GPUs × 1 micro_batch × 1 ga = 4 effective batch
```

No FSDP config needed — DDP is the default when multiple GPUs are detected by `accelerate`.

## Why This Run Will Be Different From v1.2

| | v1.2 | v1.3 (1x H200) | v1.3 (4x DDP) |
|--|------|------|------|
| seq_len | 8192 | **32768** | **32768** |
| OCR sequences truncated | ~32,500 (50%) | **0** | **0** |
| Total steps (estimated) | ~23,056 | **~5,800** | **~5,800** |
| Step time | ~8.5s uniform | **~8.5s** | **~8.5s** |
| Peak VRAM (estimated) | 45GB | **~90-120GB** | **~90-120GB/GPU** |
| GPUs | 1 | 1 | **4** |
| Total training time | ~54hrs | **~50-55hrs** | **~13-15hrs** |
| Cost | ~$105 | **~$100-110** | **~$105-120** |

v1.2 taught us that step times are roughly uniform (~8.5s) because truncation made all sequences cluster near seq_len. At 32768, step times will be different but the lesson holds: budget for uniform step times.

With 4x DDP, training time drops to ~13-15hrs at similar total cost. The GPU parallelism directly divides wall-clock time.

**Critical:** Run the dry run (20 steps) BEFORE launching full training to verify VRAM doesn't OOM at 32768. If it OOMs, fall back to `sequence_len: 16384`.

| Parameter | Value (1x GPU) | Value (4x DDP) |
|-----------|-------|-------|
| **Framework** | Axolotl 0.16.0 | Axolotl 0.16.0 |
| **Base model** | `Qwen/Qwen3.5-9B` | `Qwen/Qwen3.5-9B` |
| **Method** | LoRA (r=64, alpha=32) | LoRA (r=64, alpha=32) |
| **Sequence length** | **32768** | **32768** |
| **Sample packing** | true | true |
| **Micro batch size** | 1 | 1 |
| **Gradient accumulation** | 4 | **1** |
| **Effective batch size** | 4 | **4** (1 × 1 × 4 GPUs) |
| **Learning rate** | 1e-4 (cosine) | 1e-4 (cosine) |
| **Epochs** | 1 | 1 |
| **Precision** | BF16 | BF16 |
| **Attention** | SDPA | SDPA |
| **group_by_length** | true | true |
| **Parallelism** | None (single GPU) | **DDP** (default with multi-GPU) |
| **Hardware** | 1x H200 141GB | **4x H200 141GB** |
| **Dataset** | 230K rows | 230K rows |
| **Estimated steps** | ~5,800 | ~5,800 |
| **Estimated training time** | ~50-55 hours | **~13-15 hours** |
| **Estimated cost** | ~$100-110 | ~$105-120 |

**Reference:** OmniCoder (Tesslate) used identical setup: Qwen3.5-9B, LoRA r=64 alpha=32, Axolotl, 4x H200 DDP, sample packing. Trained 425K rows successfully.

### Important Notes on group_by_length at 32768

With `group_by_length: true` and `sequence_len: 32768`:
- OCR sequences (avg 8,477 tokens) will be packed together — multiple per window
- Previously at 8192, each OCR sequence filled its own window
- At 32768, ~3-4 OCR sequences fit per window = dramatically fewer OCR-territory slow steps
- The "slow phase" will be much shorter than v1.2's ~57% of steps
- CoderForge trajectories (which can be 4K-16K tokens) will also pack better

**Eval sub-progress bars** (the 1884-step inner bars seen in v1.2): These show the eval dataset being scored. With `evals_per_epoch: 4` and `val_set_size: 0.02`, the eval set has ~4,600 rows. Packed at 32768, the eval bar will show fewer steps than v1.2's 1884 — this is expected.

## Execution Steps

### Step 0: Provision instance and verify environment
```bash
# Option A: Multi-GPU (preferred)
# Search for available multi-GPU instances:
vastai search offers 'gpu_name=H200 num_gpus=4 dph<10.0 reliability>0.95' -o 'dph'
vastai search offers 'gpu_name=H200 num_gpus=2 dph<5.0 reliability>0.95' -o 'dph'

# Option B: Single GPU (fallback)
vastai search offers 'gpu_name=H200 num_gpus=1 dph<2.0 reliability>0.95' -o 'dph'

# After provisioning, SSH in and verify:
ssh -T -o StrictHostKeyChecking=no -p <PORT> root@<IP> "nvidia-smi | head -15"
# Confirm GPU count and VRAM

# Set HF_TOKEN in ~/.bashrc
ssh -T -o StrictHostKeyChecking=no -p <PORT> root@<IP> \
    "echo 'export HF_TOKEN=<your_rw_token>' >> ~/.bashrc"
```

### Step 1: Sync v1.3 folder, pretokenize, verify disk space
```bash
# Sync v1.3 folder from local
rsync -avP -e "ssh -T -o StrictHostKeyChecking=no -p <PORT>" \
    v1.3/ root@<IP>:/workspace/DeltaCoder/v1.3/

# Sync dataset (if not already on instance)
rsync -avP -e "ssh -T -o StrictHostKeyChecking=no -p <PORT>" \
    data/v1.2_sft_train.jsonl root@<IP>:/workspace/DeltaCoder/data/

# Run pretokenize on instance
ssh -T -o StrictHostKeyChecking=no -p <PORT> root@<IP> \
    "cd /workspace/DeltaCoder && python v1.3/scripts/pretokenize.py \
    data/v1.2_sft_train.jsonl /dev/shm/train_tokenized_v1.3.jsonl 1"

# Verify disk space
ssh -T -o StrictHostKeyChecking=no -p <PORT> root@<IP> "df -h /workspace"
```

### Step 2: Adjust config for GPU count
```bash
# For multi-GPU: set gradient_accumulation_steps to maintain effective batch of 4
# 4 GPUs: ga=1 (4 GPUs × 1 micro_batch × 1 ga = 4 effective)
# 2 GPUs: ga=2 (2 GPUs × 1 micro_batch × 2 ga = 4 effective)
# 1 GPU:  ga=4 (1 GPU  × 1 micro_batch × 4 ga = 4 effective) ← default in config

# Edit on remote if needed:
ssh -T -o StrictHostKeyChecking=no -p <PORT> root@<IP> \
    "sed -i 's/gradient_accumulation_steps: 4/gradient_accumulation_steps: 1/' \
    /workspace/DeltaCoder/v1.3/configs/axolotl.yaml"
```

### Step 3: Dry run — MANDATORY before full training
```bash
ssh -T -o StrictHostKeyChecking=no -p <PORT> root@<IP>
cd /workspace/DeltaCoder

# 20 steps to verify VRAM at 32768 (DDP auto-detected by accelerate)
accelerate launch -m axolotl.cli.train v1.3/configs/axolotl.yaml \
    --max_steps=20 2>&1 | tee logs/v1.3_dryrun.log

# Check peak memory in output:
grep "memory/max_active" logs/v1.3_dryrun.log | tail -5

# Verify DDP is active (multi-GPU only):
grep -i "distributed\|ddp\|world_size\|num_processes" logs/v1.3_dryrun.log | head -5
```

**Expected (per GPU):** peak active VRAM ~90-120GB on H200. If >135GB → OOM risk, fall back to `sequence_len: 16384`.

**If OOM:** Edit `v1.3/configs/axolotl.yaml`, change `sequence_len: 32768` to `sequence_len: 16384`, re-run dry run.

**If DDP crashes with GDN/fla errors:** Fall back to single GPU. Tear down multi-GPU instance (~$2 lost).

### Step 4: Full training run
```bash
cd /workspace/DeltaCoder
nohup accelerate launch -m axolotl.cli.train v1.3/configs/axolotl.yaml \
    > logs/v1.3_axolotl_train.log 2>&1 &

# Confirm it started
sleep 10 && tail -20 logs/v1.3_axolotl_train.log
```

### Step 5: Monitor training
```bash
# Check progress (outer bar = overall, inner bar = eval pass):
ssh -T -o StrictHostKeyChecking=no -p <PORT> root@<IP> \
    "tail -10 /workspace/DeltaCoder/logs/v1.3_axolotl_train.log"

# Key metrics to watch:
# - loss: expect 0.50-0.55 range (same as v1.2)
# - ppl: expect 1.65-1.72
# - memory/max_active: should stabilize ~90-120GB per GPU
# - step time: varies by GPU count and seq_len
#   1x H200: ~8-10s/step
#   4x H200 DDP: ~8-10s/step but 4x data per step
```

**Checkpoints:** Saved at 50% and 100% epoch (`saves_per_epoch: 2`).
**Eval:** Runs at 25%, 50%, 75%, 100% (`evals_per_epoch: 4`).

```bash

python3 -c "
from peft import PeftModel
from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

print('Loading base model...')
base = AutoModelForCausalLM.from_pretrained(
    'Qwen/Qwen3.5-9B',
    torch_dtype=torch.bfloat16,
    device_map='cpu',
    trust_remote_code=True,
    attn_implementation='eager',
)

model = PeftModel.from_pretrained(base, '/workspace/DeltaCoder/outputs/deltacoder-9b-v1.3')
print('Merging...')
model = model.merge_and_unload()
print('Saving...')
model.save_pretrained('/workspace/merged_v1.3')
AutoTokenizer.from_pretrained('Qwen/Qwen3.5-9B', trust_remote_code=True).save_pretrained('/workspace/merged_v1.3')
print('Done')
"
```
